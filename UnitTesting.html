Unit Testing with Javascript

Basic Theory

                
They are tests thet are aimed at testing the smallest testable parts of the software 
     (Это тесты, которые нацелены на тестирование самой маленькой тестируемой части программного обеспечения)
Depending on the programming language those smallest testable parts may be methods / modules / objects 
     (В зависимости от языка программирования самые маленькие тестируемые части могут быть methods / modules / objects)
In most cases these tests are written by developers. 
     (В большинстве случаев эти тесты пишутся разработчиками)

Advantages of having unit tests

                
A good wey of making sure your code does what it's supposed to do
     (Хороший способ убедиться, что ваш код выполняет то, что должен)
Early detection of bugs in code
     (Раннее обнаружение ошибок в коде )
Significantly lower risk of new issues when adding new features or changing existing code. 
     (Значительно меньший риск возникновения проблем при добавлении новых функций или изменении существующего кода)
Great tool for debagging code that you are modifying
     (Отличный инструмент для отладки кода, который вы изменяете)
Good way of ensuring code quolity
     (Хороший способ обеспечить качество кода)
Great tool for rewriting and optimizing existing code
     (Отличный инструмент для переписывания и оптимизации существующего кода)

Disadvantages of writing unit tests

                
Slower development time
     (Более медленное время развития)
Additional skill for developers to lean
     (Дополнительные навыки для разработчиков)
Require using \ learning to use additional tools
     (Требовать использования \ обучения для использования дополнительных инструментов )

The F.I.R.S.T principles of unit tests

                
Fast - a developer should not hesitate to run the tests as they are slow. All of these inkluding including setup, the actual test and its tear down should execute fast (milliseconds) as you may have thousends of tests in your entire project.
     (Быстро - разработчик должен без колебаний запускать тесты, поскольку они медленные. Все эти включения, включая настройку, собственно тест и его завершение, должны выполняться быстро (миллисекунды), поскольку у вас могут быть тысячи тестов во всем вашем проекте. )
Isolated - No order-of-run dependency, do not do any more actions after the assert statement(s), preferably single logical assert.
     (Изолированный - нет зависимости от порядка выполнения, больше не выполняйте никаких действий после утверждения, предпочтительно одно логическое утверждение.)
Repeatable - Tests should return the same result if no changes to the tested code was made.
     (Повторяемость - тесты должны возвращать тот же результат, если в тестируемый код не было внесено никаких изменений)
Self-Validating - No manual inspection required to check whether the test has passed or failed. Tests should not need any manual setup.
     (Самостоятельная проверка - не требуется ручная проверка, чтобы проверить, прошел ли тест или нет. Тесты не требуют ручной настройки.)
Thorough/Timely - Tests for corner/edge boundary values. Test for exceptions and errors. Test for illegal arguments or bad inputs. Tests should be written right after the coding is done.
     (Тщательно / Своевременно - Проверяет граничные значения углов / краев. Тестируйте на исключения и ошибки. Проверьте наличие недопустимых аргументов или неправильных входных данных. Тесты следует писать сразу после завершения кодирования)
